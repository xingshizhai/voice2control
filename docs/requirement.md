## 设计原则与体验目标

### 核心原则

| 原则 | 含义 | 反例（刻意避免） |
|------|------|------------------|
| **少一步是一步** | 默认识别结束后直达「可用状态」（填入输入框或发送），不再强制「先看结果 → 再确认 → 再按发送」 | 识别后必须 Enter 才发送、多一次必点按键 |
| **像说话，不像填表** | 按住说、松手完；静音/VAD 辅助收尾；能一句话说完就不要求用户记复杂流程 | 每轮多个模态弹窗 |
| **智能默认，可配置** | 默认贴合 IDE 聊天（如 Cursor）：**松手后粘贴到焦点输入框并发送**；高级用户可改为「只粘贴不发送」或自定义快捷键 | 一种流程打天下 |
| **补救优于拦截** | 常态信任用户口述；误识别用 **撤销、清空、重录** 解决；**高危操作**（若后续扩展）才可选二次确认 | 所有指令同等「先确认再发」 |
| **状态可感知** | 录音中、识别中、已发送有轻量反馈（托盘、短提示音、光标处变化），但不挡视线 | 长时间无反馈 |

### 体验目标（可验收）

- **主路径步数**：从「开口」到「对方收到」——在默认配置下，用户除 **按住/松开快捷键** 外 **不必** 为「发送」再按额外键（除非用户显式选择「仅粘贴」模式）。
- **学习成本**：首次使用 **3 分钟内** 能完成一轮「语音提问 AI」。
- **心流**：单次语音输入从松手到文本进入目标应用，主观等待可接受（见 NFR-1）。

## 项目背景

### 现状分析

在 **Cursor** 等 IDE 中使用 AI 助手时，用户常需在聊天框中输入中文、长句与技术术语；同时使用 OpenClaw、Claude Code、OpenCode 等工具时，也存在大量键盘指令输入。纯键盘在长句、双手占用、移动场景下效率低。

### 问题陈述

**核心问题**：与 AI 协作时，**文本输入方式拖慢节奏**；若语音方案本身步骤多、确认多，则比打字更烦。

**用户痛点**：

1. 长提示词、中文混排技术词，键盘输入慢、易错。
2. 调试、看文档时双手不便一直放在键盘上。
3. **语音工具若设计差**：识别后还要确认、再按发送，打断心流，**不如直接打字**。

### 项目目标

开发**语音输入中间件**，以**好用**为第一约束：

- 语音替代键盘输入（**优先场景：IDE 内与 AI 对话，如 Cursor 聊天框**）。
- 支持中文与技术术语。
- **默认低阻力路径**：松手即完成识别与投递（具体「仅粘贴 / 粘贴并发送」可配置）。
- 通过 **撤销、重录、可选高危确认** 管理误操作，而非默认「全员确认」。
- 兼容多种 AI 编程助手与投递方式（剪贴板、快捷键聚焦目标窗口等）。

## 目标用户

### 主要用户群体

| 用户类型 | 特征 | 需求 |
|---------|------|------|
| **IDE 开发者** | 常用 Cursor 等与 AI 对话 | 语音直接当「打字」，少按键 |
| **全栈/后端** | 使用 OpenClaw、CLI 类助手 | 长命令口述 |
| **技术写作/学生** | 大段说明、提问 | 连续口述，必要时只粘贴不发送以便编辑 |
| **无障碍用户** | 键盘使用困难 | 低步骤语音链路 |

### 用户画像（节选）

**张三 - Cursor 重度用户**

- 在 Cursor 聊天里用中文描述需求；希望 **按住快捷键说话，松手后就像自己打完字发出去一样**，不要每句再确认一遍。

**李四 - 技术博主**

- 需要大段口述；可能选择 **只粘贴到输入框不自动发送**，便于删改后再发。

## 使用场景

### 场景 A：Cursor 中与 AI 对话（优先场景）

**用户**：光标在 Cursor 聊天输入框，按住快捷键口述一整段需求。

**期望流程（默认配置）**：

```
1. 按住 F8（可配置）→ 开始录音，有轻量「录音中」反馈
2. 口述完毕 → 松开 F8 → 结束录音
3. 自动 ASR → 文本写入剪贴板并粘贴到当前焦点（聊天输入框）
4. 默认：自动发送（模拟快捷键或 Enter，以不破坏用户编辑器习惯的方式可配置）
5. 若识别错了：用户立即 Ctrl+Z（撤销输入/消息，与 IDE 行为对齐）或快捷键「重录」
```

**不要求**：识别后单独一页「请确认」再 Enter 才能发（可作为**可选「谨慎模式」**，非默认）。

---

### 场景 B：代码生成 / 工具链（OpenClaw 等）

**用户**：口述复杂需求（如带正则、多文件约束）。

**期望流程**：与场景 A 相同的主路径——**松手 → 识别 → 投递到当前焦点或约定目标**；历史记录与撤销用于纠错。

---

### 场景 C：多轮与修正

- 连续多轮提问依赖 **IDE/助手自身会话上下文**，本产品聚焦 **每一轮输入要极顺**。
- 口误当场：**重录快捷键**、或 **说取消词丢弃本轮识别结果**（在识别完成前或缓冲窗口内，见 FR-6）。

---

### 场景 D：误触与隐私

- **误录音**：仅在有按键/明确热键时开始录音，避免环境音误触。
- **说错内容**：撤销、清空输入框、重录；必要时用户开启「谨慎模式」再看一眼再发。

## 功能需求

### FR-1：语音采集（低延迟、可预期）

| 属性 | 说明 |
|------|------|
| **ID** | FR-1 |
| **名称** | 语音采集 |
| **描述** | 按住开始、松开结束；可与 VAD/静音检测配合 |
| **优先级** | P0 |
| **验收标准** | 按住即录、松开即停；最长时长与静音策略可配置；音质满足 ASR |

**详细需求**：

- 默认 **按住快捷键录音，松开结束**（默认 F8，可改）。
- 最长单次录音建议 **60 秒**（可配置）。
- 支持 **静音自动结束**（如连续 1.5 秒无语音则结束，可配置），减少「必须松手」的僵硬感（可选策略）。
- 建议格式：**16 kHz、16-bit、单声道**（与 ASR 对齐）。

---

### FR-2：语音识别（准、快）

| 属性 | 说明 |
|------|------|
| **ID** | FR-2 |
| **名称** | 语音转文字 |
| **描述** | 调用 ASR，中文与技术词场景优化 |
| **优先级** | P0 |
| **验收标准** | 准确率与响应时间见 NFR；识别结果可被下游直接投递 |

**详细需求**：

- 默认对接**局域网 ASR 服务**（WebSocket，协议见 `asr-service-api.md`）；可扩展云端 ASR。
- 中文优先，兼顾英文术语与常见技术词。
- **可选**：流式识别、边录边出字（阶段二/三，用于进一步「像打字」）；首版可整段识别。

---

### FR-3：低阻力投递与模式（产品核心）

| 属性 | 说明 |
|------|------|
| **ID** | FR-3 |
| **名称** | 低阻力投递 |
| **描述** | 识别完成后 **自动**将文本送达目标，**默认不要求**单独「确认屏」与「再按发送」 |
| **优先级** | P0 |
| **验收标准** | 默认配置下，用户完成一轮语音输入的主动操作仅为 **按住与松开快捷键**（发送由产品自动完成；若用户选择「仅粘贴」模式则例外） |

**详细需求**：

1. **默认模式（推荐）**：识别完成 → 文本进入系统剪贴板 → **粘贴到当前系统焦点控件** → **自动发送**（通过可配置的「发送」快捷键序列，适配 Cursor 等；需文档说明各 IDE 推荐配置）。
2. **仅粘贴模式**：识别完成 → 只粘贴到焦点，**不**自动发送，便于长文编辑后再由用户发送。
3. **谨慎模式（可选，非默认）**：识别完成 → 短暂展示识别文本（系统托盘/小浮窗）→ 用户 **单次快捷键** 接受或丢弃；满足少数「绝不能发错」场景，**不作为主路径**。
4. 全程提供 **可感知反馈**：录音中、识别中、成功/失败（短提示，不强制模态对话框）。

---

### FR-4：目标应用与 AI 助手集成

| 属性 | 说明 |
|------|------|
| **ID** | FR-4 |
| **名称** | 应用集成 |
| **描述** | 将文本可靠送达 Cursor 聊天框及各类 AI 编程助手使用场景 |
| **优先级** | P0 |
| **验收标准** | 至少在 **Windows** 上验证 Cursor；文档给出其他 IDE/助手配置方式 |

**详细需求**：

- **优先支持**：Cursor（聊天输入框焦点 + 粘贴 + 发送）。
- **扩展支持**：OpenClaw、Claude Code、OpenCode 等（通过 **焦点窗口 + 剪贴板 + 可选快捷键** 投递）。
- 集成方式以 **剪贴板 + 键盘模拟** 为主，避免侵入式插件依赖（首版）。

---

### FR-5：撤销、历史与重录

| 属性 | 说明 |
|------|------|
| **ID** | FR-5 |
| **名称** | 撤销与历史 |
| **描述** | 误识别、误发送后的低成本补救 |
| **优先级** | P0（与好用强相关） |
| **验收标准** | 支持撤销上一条投递/最近文本；支持重录快捷键 |

**详细需求**：

- 本地记录最近若干条识别/发送文本（如 **50 条**，可配置）。
- **全局快捷键**：撤销上一条（可与 IDE Ctrl+Z 协同说明）、**重录**（丢弃当前轮次结果再录）。
- 支持从历史选择一条 **重新投递**（P1）。

---

### FR-6：口述控制（取消 / 重录）

| 属性 | 说明 |
|------|------|
| **ID** | FR-6 |
| **名称** | 口述控制词 |
| **描述** | 在识别完成前后，用简短口令取消或重来 |
| **优先级** | P1 |
| **验收标准** | 可配置词表；不误伤正常技术对话（策略需 tuning） |

**详细需求**：

- 建议词：**取消、不对、算了、重录** 等——用于 **丢弃本轮或触发重录**（实现上可与「识别结果后处理」结合，避免与正文混淆）。
- **不再**将「确认、是的、执行」作为默认发送必要条件；若存在谨慎模式，可作为该模式下的确认词（可选）。

---

### FR-7：多轮对话与上下文

| 属性 | 说明 |
|------|------|
| **ID** | FR-7 |
| **名称** | 会话上下文 |
| **描述** | 多轮对话由 **IDE/助手** 维护；本产品不重复造轮子 |
| **优先级** | P2 |
| **验收标准** | 连续多轮语音输入体验顺畅即可 |

**详细需求**：

- 不在首版强制实现独立「会话记忆」；若未来与特定助手 API 对接，再扩展。

## 非功能需求

### NFR-1：性能要求

| 指标 | 要求 |
|------|------|
| **录音起停延迟** | 主观无感，技术目标 **<100 ms** |
| **识别时间** | 常见句长 **<3 s**（局域网 ASR） |
| **从松手到文本出现在目标处** | **<5 s**（默认路径，含粘贴与发送） |

---

### NFR-2：准确率要求

| 场景 | 目标 |
|------|------|
| **普通话** | >95% |
| **带口音** | >85% |
| **技术术语** | >90% |
| **一般噪音环境** | 可用（具体指标结合实测） |

---

### NFR-3：兼容性

| 项目 | 要求 |
|------|------|
| **操作系统** | Windows 10+（首版重点）；macOS / Linux 规划支持 |
| **Python** | 3.8+ |
| **ASR** | 局域网 ASR 服务（默认），可插拔 |

---

### NFR-4：安全与隐私

| 要求 | 说明 |
|------|------|
| **录音** | 默认本地缓冲；不上传公有云（除非用户显式配置云端 ASR） |
| **传输** | 与 ASR 服务使用 **WS / WSS**（按部署选择） |
| **历史** | 本地存储，可一键清除 |
| **权限** | 最小权限；热键与辅助功能权限按系统要求申请 |

---

### NFR-5：可用性（好用相关）

| 要求 | 说明 |
|------|------|
| **启动** | 常驻托盘/后台，**<3 s** 可开始首次录音 |
| **学习成本** | 默认路径 **3 分钟内** 完成首次成功对话 |
| **可配置** | 热键、仅粘贴/粘贴并发送、谨慎模式、各 IDE 发送键序 |
| **错误提示** | 失败时明确「ASR 不可用 / 焦点不在输入框 / 快捷键冲突」等 |

## 风险评估

### 技术风险

| 风险 | 缓解 |
|------|------|
| 识别错误导致错误发送 | **撤销、仅粘贴模式、谨慎模式**；高危场景再二次确认（后续） |
| 焦点不在聊天框 | 投递前检测/提示；文档说明「先聚焦输入框」 |
| 各 IDE 发送快捷键差异 | **可配置发送键序** + 文档化模板 |

### 产品风险

| 风险 | 缓解 |
|------|------|
| 默认自动发送「太激进」 | **仅粘贴**与**谨慎模式**作为显式选项；默认仍坚持少步骤 |
| 用户不信任语音 | 首版把 **识别结果短暂展示**（非阻塞，如托盘一行字）可选开启 |

## 验收标准

### 体验验收（优先）

- [ ] 默认配置下，用户**不必**在每次识别后单独「确认再发送」即可完成向 Cursor（或配置的目标）发送一条语音指令。
- [ ] 提供 **仅粘贴不发送** 开关。
- [ ] 提供 **谨慎模式**（可选），且默认关闭。
- [ ] 误操作可用 **撤销/重录** 在合理步数内补救。

### 功能验收

- [ ] 按住快捷键录音，松开结束；反馈清晰。
- [ ] 中文与技术场景识别可用（指标见 NFR-2）。

### D. 实现级规格：推荐默认配置（可改）

以下为 **Windows + 首版** 的默认意图；实际快捷键以用户本机 IDE 设置为准，程序只负责「按配置依次模拟按键」。**Cursor 会随版本变更快捷键**，需在设置中核对并同步到本工具配置。

#### D-1. 投递模式

| 模式 | 行为 | 默认 |
|------|------|------|
| `paste_and_send` | 将识别结果写入剪贴板 → 模拟 **粘贴** → 模拟 **发送** | ✅ 默认 |
| `paste_only` | 仅 **粘贴**，不发送 | 可选 |
| `review`（谨慎模式） | 识别结果先显示在小窗/托盘，用户单次确认后再执行粘贴/发送 | 默认关闭 |

#### D-2. 按键序列（抽象）

「发送」在多数 IDE 聊天输入框中等价于 **Enter**；「粘贴」为 **Ctrl+V**（macOS 为 **Cmd+V**，跨平台实现时需按 OS 映射）。

| 步骤 | 含义 | 典型键（Windows） |
|------|------|---------------------|
| `pre_focus_chat`（可选） | 将焦点切到 AI 聊天输入区 | 由用户在 IDE 中查看后填写，例如 Cursor 常见为打开/聚焦 Chat 的全局快捷键（**以 Cursor 设置 ▸ Keyboard Shortcuts 为准**） |
| `paste` | 从剪贴板插入文本 | `ctrl+v` |
| `send` | 提交当前输入 | `enter` |

**首版推荐用户操作**：先手动 **单击** 聊天输入框或使用 IDE 自带快捷键 **聚焦** 到输入框，再使用语音快捷键；可减少「焦点不在聊天框」导致的误粘贴。后续版本可加强焦点检测或自动化 `pre_focus_chat`。

#### D-3. 预设配置档（示例名）

| 配置档 ID | 说明 | `pre_focus_chat` | `paste_and_send` 序列 |
|-----------|------|------------------|------------------------|
| `cursor_win` | Cursor（Windows），用户已聚焦输入框 | 空 | `paste` → `send` |
| `cursor_win_auto` | 尝试先聚焦 Chat 再粘贴发送（**需用户填入与 IDE 一致的快捷键**） | 用户自定义 | `pre_focus_chat` → `paste` → `send` |
| `paste_only_generic` | 仅粘贴到当前焦点 | 空 | `paste` |

> **注意**：若输入法处于中文状态，部分 IDE 对 `Enter` 的行为可能不同；建议在语音输入前切换到英文输入法或确认 Enter 会提交消息（可在产品说明中提示）。

### E. 配置文件草案（与实现对齐）

首版建议使用 **YAML**，与仓库内 `config.example.yaml` 同步演进。

| 键 | 类型 | 说明 |
|----|------|------|
| `asr.base_url` / `asr.ws_path` | string | ASR 服务 WebSocket 地址与路径 |
| `hotkey.push_to_talk` | string | 按住说话，如 `f8` |
| `delivery.mode` | enum | `paste_and_send` / `paste_only` / `review` |
| `delivery.actions` | list | 有序动作：`pre_focus_chat` / `paste` / `send` 及对应键名 |
| `delivery.profile` | string | 选用预设档，可覆盖部分 `actions` |
| `history.max_items` | int | 本地历史条数上限 |

具体字段以 `config.example.yaml` 为准。
